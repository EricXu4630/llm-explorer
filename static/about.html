How LLM Explorer Works

LLM Explorer is a deliberately thin harness around three LLM provider APIs. This page documents exactly what the harness does, what each API handles natively, how memory and Skills work, and where every file lives.

The harness does almost nothing. Its entire job is:

  1. Pass the conversation + tools config to the provider's API
  2. Stream events back to the browser
  3. Execute three client-side tools (memory, bash, bash_tool)
  4. Save container IDs and skill IDs to disk for reuse

What the Harness Does client-side #

🧠 memory_20250818 (Anthropic)
The model calls this tool with filesystem commands (view, create, str_replace, insert, delete, rename). The harness maps /memories/ → memories/ on your disk and executes the operation. No API call is made. This is how the model writes durable memory.
🖥️ bash (Gemini) + bash_20250124 (Anthropic)
The model sends a command string. The harness runs it in workspace/ via Python subprocess (Windows) or real bash (Unix). Results are returned to the model. On Unix, full bash. On Windows, a Python emulator handles common commands (ls, cat, echo, grep, etc.)
Everything else is handled by the API. Web search, code execution, URL fetching, image generation, MCP calls, computer use — all of it runs on the provider's infrastructure. The harness never makes any of those calls.

AGENTS.md & Memory #

There are two separate files called AGENTS.md. They serve different purposes and neither replaces the other.
📄 AGENTS.md — Instructions (static)
Lives at llm-explorer/AGENTS.md.

Loaded as the system prompt on every API call to all three providers. Defines the agent's role, memory protocol, tool list, and behavioral rules.

Never written by the model. You edit this file manually to change how the agent behaves.
🧠 memories/AGENTS.md — Persistent Memory (dynamic)
Lives at llm-explorer/memories/AGENTS.md.

Written by the model via the memory_20250818 tool (Anthropic only). The model reads it at the start of every session to restore context from prior conversations.

Contains: learned user preferences, durable facts, VIP senders, recurring rules.

Skills (SKILL.md Standard) #

Skills follow the open agentskills.io standard. Each skill is a folder with a SKILL.md file containing a YAML frontmatter header and instructions. Browse community skills at skillsmp.com.

What happens when you load a skill

1. Upload once
Harness zips SKILL.md, uploads to provider's Skills API, caches the skill_id. Only happens on first load.
2. Reference by ID
Every API call includes only the skill_id. The SKILL.md content is never in the request.
3. API does the rest
Provider loads skill metadata first (~100 tokens), then full SKILL.md only when the model decides it's relevant.

Skill handling by provider

✅ Anthropic — Native Skills API
Skills uploaded via /v1/skills.
Referenced in container.skills parameter.
Requires code_execution tool + betas:
code-execution-2025-08-25, skills-2025-10-02

Progressive disclosure is handled entirely server-side in Anthropic's container. The model literally reads SKILL.md from the container filesystem via bash.
✅ OpenAI — Shell Environment Skills
Skills uploaded via client.skills.create().
Referenced in tools[shell].environment.skills.
Mounted in the hosted shell container.

The model can read skill files via bash commands in the container.
⚠️ Gemini — System Prompt Injection
Gemini has no native Skills API. The full SKILL.md content is injected into the system_instruction on every call. No upload, no progressive disclosure — the entire skill content counts against the context window each time.

File Structure #

All state lives on your machine. Nothing is stored in the cloud except provider-side data (container files, uploaded skills, uploaded file_ids).

llm-explorer/
├── AGENTS.md ← static instructions (system prompt for all providers)
│
├── memories/ ← model's writable memory (memory_20250818 tool)
│ ├── AGENTS.md ← model writes durable preferences here
│ ├── *.md ← any other files model chooses to create
│ └── skills/ ← skills the model has loaded
│
├── workspace/ ← bash workspace for Gemini (permanent local)
│ └── todo.md, *.py, ... ← files Gemini writes via bash tool
│
├── sessions/ ← conversation history + container IDs
│ └── default.json ← {container_ids: {anthropic: "...", openai: "..."}}
│
├── skill_ids_cache.json ← Anthropic skill_id cache (hash → skill_id)
└── openai_skill_ids.json ← OpenAI skill_id cache

MCP Servers #

MCP (Model Context Protocol) lets you connect external tools to the model. For Anthropic and OpenAI, the API makes the MCP calls — your harness just passes the server URL. For Gemini, there is no native connector.

✅ Anthropic + OpenAI — Native MCP
You provide: server URL + optional auth token.
The provider's API calls the MCP server, discovers tools, executes them.
The harness does nothing except pass the URL in the request.

Requires: publicly accessible HTTPS endpoint (SSE or Streamable HTTP). Local servers need ngrok or similar.

Free public test server: https://mcp.deepwiki.com/mcp
⚠️ Gemini — No Native MCP
Gemini's API has no MCP connector. To use MCP with Gemini you'd need to implement a client-side loop:
1. Call MCP server to discover tools
2. Convert to function_declarations
3. When model calls a function, call MCP and return result

This would require additional harness code not currently implemented.
