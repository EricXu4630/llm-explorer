What the harness does vs. what the API does

TaskWhere it runsDetail
Web searchAPIProvider executes on their servers. Harness passes tool config only.
Code executionAPIRuns in provider's sandboxed container. Harness never sees the code run.
URL fetch / mapsAPIProvider fetches and filters. Harness receives only the final answer.
Image generationAPIOpenAI only. Harness receives base64 image and renders it.
MCP tool callsAPIAnthropic + OpenAI call your MCP server directly. Harness passes the URL.
Skills (SKILL.md)APIAnthropic + OpenAI: harness uploads zip once, API handles progressive disclosure. Gemini: harness injects full content into system prompt.
Memory writesharnessModel calls memory_20250818 tool → harness writes to memories/ on disk.
Bash commandsharnessGemini bash tool + Anthropic bash_20250124 → harness runs command in workspace/.
Conversation historyharnessBrowser holds the message array and sends it on every call. The API has no memory of prior turns.
Container ID reuseharnessAfter code execution, harness saves container_id to sessions/ and passes it next call so files persist (30-day window).

AGENTS.md and memory

📄 AGENTS.md — Instructions
llm-explorer/AGENTS.md

Loaded as the system prompt on every call. Defines role, tools, behavior rules.

Never written by the model. You edit this manually.
🧠 memories/AGENTS.md — Memory
llm-explorer/memories/AGENTS.md

Written by the model via memory_20250818 (Anthropic only). Model reads it at session start to restore context.

Contains: learned preferences, facts, recurring rules.

Cross-session flow: model writes to /memories/ → harness saves to disk → next session, model runs memory view /memories → reads it back.

Skills (SKILL.md standard)

StepWhat happens
First loadHarness zips SKILL.md, uploads to provider Skills API, caches the skill_id locally.
Every call afterOnly the skill_id is sent. SKILL.md content never appears in requests again.
Progressive disclosure
(Anthropic + OpenAI)		Level 1: ~100 tokens (name + description, every call). Level 2: full SKILL.md, only when model triggers skill. Level 3: bundled resources (scripts, refs), on demand.
GeminiNo native Skills API. Full SKILL.md content injected into system prompt every call.

Local file structure

memories/ ← model's writable memory (permanent)
  AGENTS.md ← learned preferences, written by model
  *.md, *.json, ... ← whatever model creates
workspace/ ← bash sandbox for Gemini (permanent)
sessions/default.json ← container IDs (Anthropic: 30d, OpenAI: configurable)
skill_ids_cache.json ← Anthropic skill_id → content hash map
openai_skill_ids.json ← OpenAI skill_id cache
AGENTS.md ← static instructions (system prompt, you edit this)
.env ← API keys (never committed)

On Railway/Render: memories/, workspace/, and sessions/ reset on redeploy (ephemeral filesystem). Add a Postgres DB for persistence.
